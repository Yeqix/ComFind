# 组合数学公式检索应用 PRD

## 一、产品概述

### 1.1 产品名称

暂定名：**CombFind** / **CombiOEIS** / **FormulaAtlas**

---

## 1.2 产品背景

OEIS 解决的是“数列查询”问题：用户输入一个数列片段，系统返回最相近的已知数列及其定义、性质与参考资料。

在组合数学领域，大量知识以**公式、恒等式、递推关系、生成函数、计数表达式**等形式存在。研究者、学生和工程应用人员在遇到一个组合数学公式时，常常会有以下需求：

- 这个公式是否已经存在于某个已知结论中？
- 它与哪个经典公式最相近？
- 它是否是某个公式的变形、特例、推广形式？
- 是否存在重复录入或命名不同但本质相同的公式？
- 能否快速判断公式结构、验证基本合理性，并找到出处？

目前这类需求通常依赖：
- 人工查书/查论文
- 通用搜索引擎
- 数学搜索网站（但很多偏 LaTeX 文本匹配，结构化检索能力不够）
- 专家经验判断

因此需要一个**轻量、可更新、专注组合数学公式的检索系统**，支持**公式输入、规范化、AI 辅助检查、相似度匹配与结果排序**。

---

## 1.3 产品定位

本产品是一个：

- **面向组合数学公式的垂直检索工具**
- **支持 AI 辅助公式结构理解和相似公式匹配**
- **优先追求简单易用，而不是功能大而全**
- **支持数据持续更新与维护的知识库型产品**

它不是：
- 通用符号计算系统
- 完整数学证明系统
- 全领域公式百科
- 复杂的学术写作平台

---

## 1.4 产品目标

### 核心目标
让用户输入一个组合数学公式后，系统能够：

1. **快速检查公式是否规范/可解析**
2. **识别公式类型与关键结构**
3. **返回知识库中重合率/相似度最高的组合数学公式**
4. **展示相似原因与差异点**
5. **支持持续录入、修订、更新公式数据**

### 业务目标
- 建立一个可持续扩展的组合数学公式知识库
- 降低组合数学文献检索成本
- 提高公式去重、比对、复用效率
- 为后续扩展到更广数学领域打下基础

# 二、产品概念与目标用户

## 2.1 产品概念

产品核心概念可以概括为：

> **“组合数学公式版 OEIS”**
> 用户输入一个组合公式，系统通过公式解析、标准化、特征提取和 AI 相似度匹配，返回最接近的已知公式，并提供基础校验、标签分类、出处与关联信息。

---

## 2.2 目标用户

### 2.2.1 核心用户

#### 1）组合数学研究者 / 数学研究人员
需求：
- 查找某个公式是否已知
- 比对公式与已有恒等式的关系
- 避免重复发现
- 查找文献来源与经典命名

特点：
- 公式输入较复杂
- 对检索准确性和可解释性要求高

#### 2）高校教师 / 研究生 / 本科高年级学生
需求：
- 学习组合恒等式、生成函数、递推公式
- 快速定位相近经典结论
- 理解公式归属与变形关系

特点：
- 偏重教学与学习
- 需要结果直观、简单易用

#### 3）算法工程师 / 竞赛选手 / 理论计算机研究者
需求：
- 遇到计数公式时快速判断来源
- 查询二项式系数、Catalan、Stirling、Bell、分拆等相关公式
- 找到可复用表达式

特点：
- 关注检索速度与实用性
- 不一定需要严格证明，但需要高质量候选结果

---

## 2.3 次级用户

#### 4）知识库维护者 / 编辑者
需求：
- 录入新公式
- 编辑标签、解释、参考文献
- 处理重复条目与质量审核

#### 5）数学平台运营方 / 学术数据库建设者
需求：
- 作为公式知识检索模块嵌入其他系统
- 提供 API 能力

---

## 2.4 用户痛点

1. **公式难搜索**
普通搜索引擎对数学公式理解有限，LaTeX 写法不同会导致结果分散。

2. **同一公式有多种写法**
例如：
- \(\binom{n}{k} = \frac{n!}{k!(n-k)!}\)
- 用 Gamma 函数写法
- 用递推式定义
这些可能本质相关，但文本不同。

3. **组合公式经常存在“变形”等价”**
变量替换、上下标变换、求和指标调整都会使检索困难。

4. **难判断是不是已知经典结论**
尤其对学生和非本领域研究者而言，缺少快速入口。

5. **缺少结构化知识库**
公式、别名、标签、文献来源、应用场景通常分散在书籍与论文中。

# 四、详细功能列表

---

## 4.1 公式输入与解析

### 功能描述
用户可以输入组合数学公式，系统自动进行解析和预处理。

### 支持输入形式
- LaTeX 输入
- 纯文本数学表达式输入
- 可选的公式编辑器输入（MathQuill 类）
- 后续可支持图片 OCR（非 MVP）

### 处理能力
- 识别等式、求和、乘积、递推、生成函数表达式
- 识别常见组合函数符号：
  - \(\binom{n}{k}\)
  - \(S(n,k)\)（Stirling 数）
  - \(B_n\)（Bell 数）
  - \(C_n\)（Catalan 数）
  - \(P(n,k)\)、分拆函数等
- 将用户输入转为统一中间表示（AST / Symbolic Tree）

### 子功能
1. 输入校验
2. 基础语法提示
3. 公式预览
4. 解析失败提示
5. 支持公式中变量识别

---

## 4.2 公式标准化

### 功能描述
将同一公式的不同写法尽量映射到统一表示，以提高匹配准确率。

### 标准化内容
- 变量名归一化
  如 \(i,j,n,k\) 映射为标准占位符
- 空格、括号、LaTeX 宏差异消除
- 求和上下界形式规范化
- 常见等价表达转换
- 排序无关项规范化
- 公式左右两边统一标准表示
- 常见组合符号别名统一

### 示例
- \(\sum_{k=0}^{n}\binom{n}{k}=2^n\)
- \(\sum_{i=0}^{n} C(n,i)=2^n\)

经过标准化后应能识别为高相似结构。

---

## 4.3 AI 辅助公式检查

### 功能描述
在检索前，AI 对用户公式做快速检查，给出基础反馈。

### 检查项
1. **结构完整性检查**
   - 是否为合法数学表达式
   - 等式/递推/求和结构是否完整

2. **组合数学类型识别**
   - 恒等式
   - 递推关系
   - 生成函数
   - 计数公式
   - 概率组合公式
   - 分拆/排列/组合类

3. **符号合理性提示**
   - 变量可能未定义
   - 求和指标与上下界不一致
   - 左右维度或结构明显不一致（仅提示，不作严格结论）

4. **可检索性提示**
   - 当前公式适合直接检索
   - 建议简化/补全后再检索

### 输出形式
- “已成功解析”
- “识别为二项式恒等式”
- “存在未绑定变量：m”
- “建议补充求和下界”
- “已为你按标准形式检索”

---

## 4.4 相似公式检索

### 功能描述
这是产品的核心功能。系统从知识库中找到与输入公式最相近的条目，并按相似度排序。

### 检索维度
1. **结构相似度**
   - AST 树相似
   - 运算符结构相似
   - 求和/递推层级相似

2. **符号模式相似度**
   - 是否包含相同组合对象
   - 是否属于同类函数家族

3. **语义相似度**
   - 公式作用或数学意义接近
   - 例如都在描述 Catalan 数不同形式

4. **标准化后匹配度**
   - 变量替换后是否高度一致
   - 常见变形后是否接近

5. **标签与分类相关性**
   - 二项式恒等式
   - Stirling 数
   - 生成函数等

### 输出内容
- Top 1 最相似公式
- Top N 候选列表
- 每条结果显示相似度分数
- 匹配原因说明
- 与输入的差异提示

### 结果示例字段
- 公式标题
- 标准公式
- 相似度分数（如 0.93）
- 分类标签
- 公式简介
- 参考文献
- 来源链接
- 变形说明

---

## 4.5 检索结果详情页

### 功能描述
点击检索结果后，进入公式详情页。

### 展示内容

---

## 4.6 相似原因解释

### 功能描述
让检索结果具备可解释性，而不是只给一个黑盒分数。

### 解释项
- 与输入公式共享相同求和结构
- 变量替换后可基本对齐
- 都包含二项式系数与幂和结构
- 差异在于上下界或常数项
- 当前结果可能是输入公式的特例 / 推广 / 等价变形

### 目标
增强用户信任，提高学术使用场景下的可接受度。

# 六、界面设计要求

产品强调“简单易用，不要太复杂”，因此界面设计应遵循：**清晰、轻量、学术感、低学习成本**。

---

## 6.1 设计原则

1. **首页即搜索**
   - 像 OEIS 一样，打开就能输入和查
   - 不要复杂导航层级

2. **数学公式展示优先**
   - 公式渲染必须清晰
   - 支持桌面端良好阅读

3. **结果解释可见**
   - 不只是给相似度，还要说明“为什么像”

4. **减少操作步骤**
   - 输入 → 检查 → 检索 → 结果
   - 核心路径尽量在 1~2 个页面内完成

5. **后台和前台分离**
   - 普通用户界面极简
   - 管理员维护界面可稍专业化

---

## 6.2 页面结构建议

### 6.2.1 首页 / 搜索页
模块：
- 顶部 Logo + 简短产品说明

- 支持 LaTeX 输入提示
- 公式实时预览
- “检查并检索”按钮

页面特点：
- 类似搜索引擎首页

每条结果卡片显示：

# 七、技术栈建议

考虑到产品目标是“简单可用 + 便于迭代 + AI 能力接入方便”，建议采用成熟、轻量、扩展性好的技术方案。

---

## 7.1 前端技术栈

### 推荐方案
- **React / Next.js**
  - 便于快速搭建搜索型应用
  - SSR/CSR 灵活
  - 生态成熟

### UI 组件
- Tailwind CSS
- shadcn/ui 或 Ant Design（后台管理可优先 Ant Design）

### 数学公式渲染
- **KaTeX** 或 **MathJax**
  - KaTeX 更快，适合检索页面
  - MathJax 兼容性更强

### 公式输入
- 简单 MVP：
  - 普通文本框 + LaTeX 预览
- 后续增强：
  - MathQuill / 自定义公式编辑器

---

## 7.2 后端技术栈

### 推荐方案一：Python 技术栈
- **FastAPI**
  - 适合 AI、解析、检索服务
  - 接口清晰，性能较好，开发快

### 原因
- 数学公式处理、NLP、向量检索、AI 调用生态强
- 与 SymPy、PyTorch、检索库整合方便

### 备选方案
- Node.js（如果团队前后端统一 JS）
- Go（如果极度强调高性能 API）

## 7.3 数学解析与标准化

### 建议工具
- **SymPy**
  - 公式解析、符号表达、简化、标准化辅助
- 自定义 parser / LaTeX parser
  - 结合 latex2sympy 等工具
- 自定义 AST 转换层
  - 用于“组合数学结构特化表示”

### 说明
通用 CAS 不能直接解决全部组合数学等价问题，因此应采用：
- 通用解析工具
- 加上业务规则层
- 再结合 AI 语义匹配

## 7.4 检索与相似度计算

建议采用**混合检索架构**：

### 1）规则/结构检索
- 标准化后的字符串匹配
- AST 结构特征匹配
- 标签过滤

### 2）向量语义检索
- 使用 embedding 模型对公式及说明文本向量化
- 存储在向量数据库中进行近邻搜索

### 3）重排序
- 对 Top K 结果进行更精细的相似度打分
- 综合：
  - 结构相似
  - 符号匹配
  - 语义匹配
  - 标签匹配

### 向量数据库建议
- PostgreSQL + pgvector
- 或 Milvus / Qdrant

若产品早期规模不大，优先推荐：
- **PostgreSQL + pgvector**
  - 成本低
  - 简单
  - 易维护

## 7.5 数据库

### 主数据库
- **PostgreSQL**
  - 结构化数据适合
  - 支持 JSON 字段
  - 能配合 pgvector 做向量检索

### 缓存

## 7.6 AI 能力接入建议

AI 主要用于：
1. 公式类型识别
2. 检查提示生成
3. 相似原因解释
4. 结果重排序辅助

### 技术路径
- 外部大模型 API + 本地规则系统

### 原因

- 提高稳定性
- 保证结果可控
- 更适合学术检索场景

# 八、非功能需求

---

## 8.1 性能要求

### 响应时间

- 复杂公式检索：**5 秒内返回**
- 公式预览渲染：**1 秒内完成**
- 后台公式保存：**2 秒内完成**

### 并发能力
- MVP 阶段支持 100~300 并发用户
- 保证检索服务在教学或实验室场景中稳定可用

### 可扩展性
- 支持公式条目从 1 万扩展至 100 万级
- 检索架构支持向量索引与结构索引扩容

---

## 8.2 可用性要求

- 核心检索功能可用性：**99.5%+**
- 输入、检索、详情页是关键路径，优先保证稳定
- 异常时要给用户明确错误提示，不可无响应

---

## 8.3 安全要求

### 用户侧安全
- 防止恶意脚本输入
- 公式输入需做好转义与校验
- 防止 SQL 注入、XSS、CSRF

### 管理后台安全
- 管理员鉴权
- 角色权限控制
- 审计日志
- 敏感操作二次确认

### AI 调用安全
- 限制 prompt 注入风险
- 不将敏感后台信息暴露给模型
- 日志中对用户输入进行必要脱敏

---

## 8.4 数据质量要求

这是本产品极关键的一项非功能需求。

### 要求
1. 每条公式需有唯一 ID
2. 公式录入需尽量包含来源
3. 重复条目应能标记
4. 公式分类要统一
5. 重要字段不能为空：
   - 公式本体
   - 标准化表示
   - 类型
   - 来源或说明

### 质量机制
- 后台审核流程
- 重复项提醒
- 历史版本可回滚
- 数据修改留痕

---

## 8.5 可维护性要求

- 检索规则与业务逻辑模块化
- 公式标准化规则可配置
- AI 模块可替换
- 数据结构预留扩展字段
- 支持日志监控与错误追踪

---

## 8.6 可解释性要求

对学术用户而言，系统结果不能完全黑盒化。

### 要求
- 每个检索结果至少提供一条“为什么匹配”的解释
- 相似度分数应有相对稳定性
- AI 检查提示需区分：
  - 明确结构问题
  - 可能存在的问题
  - 推测性解释

## 8.7 合规与版权要求

- 对公式本身一般问题不大，但说明文本、参考内容需关注版权
- 引用书籍、论文时应存储标准引用信息而非大段转载
- 若接入第三方数据源，要确认授权方式

# 十二、风险与应对

---

## 12.1 公式等价识别困难
### 风险
数学公式存在大量变形，仅靠字符串或普通 embedding 效果有限。

### 应对
- 采用规则 + AST + 向量混合检索
- 先聚焦组合数学常见结构
- 逐步积累领域规则库

---

## 12.2 AI 解释不稳定
### 风险
AI 可能给出听起来合理但不严格的解释。

### 应对
- AI 只做“辅助检查”和“解释生成”
- 核心排序依赖规则和结构检索
- 对 AI 输出增加“可能/建议”措辞约束

---

## 12.3 知识库冷启动
### 风险
早期数据量不足，检索价值有限。

### 应对
- 先聚焦经典组合公式集
- 从教材、经典恒等式、标准序列相关公式入手
- 优先建设高频查询领域：
  - 二项式恒等式
  - Catalan
  - Stirling
  - Bell
  - 容斥
  - 生成函数

---

## 12.4 用户输入格式混乱
### 风险
LaTeX 不规范、公式残缺、符号习惯不一。

### 应对
- 做好输入示例
- 实时渲染和报错提示
- 宽容解析策略
- 提供“标准写法建议”

---
